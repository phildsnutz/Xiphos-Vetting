# Helios Graph State Contract

## Purpose

Helios needs a hard contract for graph truth states so inference does not poison observed memory.

Every important graph edge or claim must sit in one of four states:

1. `observed`
2. `inferred`
3. `predicted`
4. `reviewed`

## State Definitions

### `observed`

Definition:
- supported directly by source evidence
- tied to provenance, source, and time

Allowed examples:
- `subsidiary_of`
- `contracts_with`
- `officer_of`
- `sanctioned_on`

Requirements:
- evidence artifact or source citation
- confidence based on source quality, not model belief
- timestamp or freshness metadata when available

### `inferred`

Definition:
- derived from structural reasoning or rule application over observed claims

Examples:
- indirect sanctions exposure
- likely FOCI pressure through control chain
- inferred ecosystem dependency

Requirements:
- inference method recorded
- parent observed claims recorded
- support and confidence recorded
- never silently merged into observed truth

### `predicted`

Definition:
- generated by embeddings or temporal models as a likely but unverified missing link

Examples:
- likely future subcontractor link
- likely hidden owner path
- likely recompete challenger

Requirements:
- model family recorded
- prediction score recorded
- feature or neighborhood context recorded
- analyst-visible caveat required

### `reviewed`

Definition:
- an analyst has reviewed an inferred or predicted item and either promoted, held, or rejected it

Substates:
- `reviewed_promoted`
- `reviewed_hold`
- `reviewed_rejected`

Requirements:
- reviewer identity
- review timestamp
- rationale

## Promotion Rules

### Observed

- may feed inference directly
- may appear in client deliverables without caveat if source tier and contradiction state allow it

### Inferred

- may appear in client deliverables only with explicit inferential framing
- may become `reviewed_promoted` after analyst acceptance
- must never overwrite the underlying observed claims

### Predicted

- internal analyst use by default
- must not appear in client deliverables without explicit review and framing
- must not auto-promote into observed graph memory

### Reviewed

- `reviewed_promoted` can influence downstream ranking and War Room prioritization
- even when promoted, original provenance must remain visible

## UI Rules

### Front Porch

- never expose the state model directly
- use graph-backed reasoning to ask better questions and disambiguate entities
- do not show raw prediction jargon

### War Room

- show the distinction between observed, inferred, predicted, and reviewed
- let analysts challenge or promote graph findings

### Returned Brief

- observed and reviewed-promoted findings can appear cleanly
- inferred findings need disciplined caveats
- predicted findings stay out unless explicitly framed as forward-looking assessment

## Backend Implications

The state contract should be represented in:

- knowledge graph persistence
- predicted-link review queue
- rule mining outputs
- temporal model outputs
- dossier generation
- AXIOM gap closure promotion hooks

## Immediate Priority

Before large-scale embeddings:

1. treat graph memory as first-class in entity discovery
2. preserve observed vs inferred vs predicted boundaries
3. expose state cleanly in War Room
