# HELIOS_WAR_ROOM_DOCTRINE_2026-04-03

## Bottom line

War Room is not a dashboard.
War Room is not admin mode.
War Room is not “advanced settings.”

War Room is the deeper room where AXIOM and a sophisticated user work the problem together.

The user should feel:

- they walked into a more serious space
- the problem is now live in front of them
- AXIOM is still leading
- the evidence is visible
- the tooling supports the conversation instead of replacing it

If Front Porch is:

- “brief me”

then War Room is:

- “work with me”

---

## Why War Room matters

Front Porch is the volume surface.
It gets the user from question to framed problem with minimal friction.

War Room exists because some users and some problems require:

- ambiguity handling
- trail exposure
- challenge and redirection
- collection steering
- peer-level collaboration

Without War Room, Helios risks feeling shallow to practitioners.
With the wrong War Room, Helios risks feeling like a generic intelligence dashboard.

The doctrine here is designed to avoid both failures.

## Research and reference synthesis

The visual references imply a consistent set of lessons:

### Reference lesson 1: Dark spaces can feel premium and serious

The dark visual language works when:

- one central problem dominates
- auxiliary surfaces orbit it
- the visual field feels spatial, not grid-bound

This is good for Helios because War Room should feel like a more consequential room than Front Porch.

### Reference lesson 2: The center should be the work, not the chrome

The strongest references do not use the center for navigation.
They use the center for:

- active creation
- active interpretation
- active response

For Helios, that means the center should be:

- the conversation
- the finding
- the active line of inquiry

Not:

- menu systems
- dashboards
- static KPI blocks

### Reference lesson 3: Auxiliary panes should sharpen the main thread

The surrounding panes work when they:

- support the central problem
- expose useful detail
- remain subordinate to the core task

For Helios:

- source trails
- graph paths
- target summaries
- collection leads
- alert context

must all support the live inquiry.

### Reference lesson 4: A command-center mood is powerful but dangerous

The third reference shows the emotional upside of a command room:

- urgency
- seriousness
- situational awareness

But Helios should borrow that selectively.
If overdone, it becomes:

- theatrical
- militarized
- noisy
- game-like

War Room should feel elite and focused, not cosplay.

## The emotional contract

War Room must create these feelings:

### 1. Entering a sharper room

The user should feel:

- this is the same problem, but at a higher fidelity
- they are now inside the work
- the system did not switch products on them

### 2. Shared seriousness

The user should feel:

- AXIOM is not summarizing for beginners anymore
- it is thinking with them
- the conversation has more depth and more edge

### 3. Control without burden

The user should feel:

- they can redirect or challenge the work
- but they are still not required to operate machinery

### 4. Evidence with purpose

The user should feel:

- every visible panel matters
- each one earns its right to exist
- nothing is decorative

### 5. Calm command

The user should feel:

- this is more intense than Front Porch
- but still composed
- still trustworthy
- still restrained

## War Room purpose

War Room exists to support:

- ownership-chain tracing
- incumbent / teammate analysis
- vulnerability analysis
- dossier gap closure
- source challenge
- provenance review
- graph path validation
- collection redirection

War Room should not be required for:

- normal dossier delivery
- simple vendor assessment
- light vehicle lookups
- casual product exploration

If a user can finish in Front Porch, that is a success.
War Room is a deeper room, not an upsell tax.

---

## War Room visual identity

War Room should differ visibly from Front Porch.

### Front Porch identity

- bright or softly dark
- spacious
- quiet
- almost no visible machinery

### War Room identity

- dark
- immersive
- lower ambient brightness
- tighter contrast
- more depth
- more surface layering

This contrast matters because it signals:

- different task
- different altitude
- same product

### War Room palette direction

The palette should be:

- charcoal / deep slate / near-black foundation
- restrained cool highlights
- one strong accent family
- semantic color only when meaningfully tied to evidence or status

Avoid:

- rainbow dashboard palettes
- decorative gradients fighting the content
- multiple accent systems

### War Room lighting model

The best War Room feel comes from:

- darker field
- brighter active surfaces
- subtle glows only where attention is deserved

The visual hierarchy should read:

1. active conversation / active finding
2. evidence and trail
3. contextual controls
4. ambient chrome

## War Room anatomy

War Room should be spatial, but disciplined.

### Core rule

The center belongs to the problem.

### Recommended anatomy

#### Center pane

The center pane is the live intelligence conversation.

It should contain:

- AXIOM dialogue
- user replies
- live finding cards
- active line-of-inquiry state
- dossier fragments when relevant

This is the main surface.
It is not a side widget.

#### Left context pane

This pane holds the stable frame around the problem.

It may include:

- target summary
- engagement title
- object type
- current mission framing
- current anchors
- working assumptions
- current stance

This pane should answer:

- what are we working?
- why does it matter?
- what context is already fixed?

#### Right evidence pane

This pane holds dynamic supporting intelligence.

It may include:

- source trail
- evidence snippets
- source-tier badges
- alerts
- active gaps
- collection leads
- provenance chain summaries

This pane should answer:

- why does AXIOM believe this?
- what is still weak?
- what changed?

#### Optional contextual subpanes

These should appear only when relevant:

- graph path inspector
- timeline or event stream
- challenge form
- alternate hypothesis panel
- dossier artifact viewer

These are conditional surfaces, not permanent chrome.

## Pane hierarchy rules

### Rule 1: Conversation stays primary

If the conversation no longer feels central, War Room has failed.

### Rule 2: Evidence precedes control

The user should see:

- the finding
- the trail
- the implication

before they see:

- controls
- toggles
- settings

### Rule 3: The graph is subordinate

The graph must not dominate the room visually unless the graph itself is the active investigative object.

The graph is:

- an explainer
- a path surface
- a trail viewer

It is not:

- the hero object by default

### Rule 4: Fixed chrome should be minimal

War Room may have more structure than Front Porch, but it still should not drown the user in persistent chrome.

Avoid:

- giant nav rails
- multi-row toolbars
- global control strips
- equal-weight pane framing

## AXIOM behavior in War Room

AXIOM should speak differently here.

Not differently in personality.
Differently in altitude.

### Front Porch AXIOM

- clarifies
- frames
- reassures through competence
- compresses

### War Room AXIOM

- exposes its trail
- asks sharper questions
- offers concrete next moves
- invites challenge
- narrates ambiguity precisely

### Good War Room language

- “I traced the ownership chain through three Delaware LLCs and hit a wall at a Cayman entity.”
- “The public filings do not prove control, but the lien pattern suggests a lender with effective leverage.”
- “I can keep pulling on the UCC thread or pivot to teaming residue from the incumbent vehicle. Which matters more to you right now?”

### Bad War Room language

- “Connector 17 returned 4 records.”
- “Validation gate accepted claim with score 0.74.”
- “Graph sync complete.”

Those may be system truths.
They are not user-facing intelligence language.

## War Room interaction model

War Room should operate through four kinds of moves:

### 1. AXIOM surfaces a finding

Example:

- “Amentum’s incumbent advantage looks real, but the subcontractor picture is unusually dark.”

### 2. AXIOM shows the trail

Example:

- “That conclusion is coming from amendment history, teammate residue, and lack of visible subaward reporting.”

### 3. AXIOM offers a focused next move

Example:

- “I can now pull on protest history, archived amendments, or likely teammate inference.”

### 4. The user responds naturally

Example:

- “Work the teammate inference first.”

This is the core interaction loop.

Not:

- the user hunting through controls
- the user configuring processes
- the user assembling the environment manually

## War Room modes

War Room should support several interaction modes, but expose them softly.

### Mode 1: Trail mode

Used when the user wants to understand why AXIOM believes something.

Primary surfaces:

- conversation
- evidence trail
- provenance chain

### Mode 2: Collection mode

Used when the user wants to redirect or deepen the work.

Primary surfaces:

- conversation
- gaps
- collection leads
- optional source suggestions

### Mode 3: Graph mode

Used when the graph path itself is the object of interest.

Primary surfaces:

- conversation
- graph path
- node / edge provenance

### Mode 4: Delivery mode

Used when the user wants to shape the dossier or final narrative.

Primary surfaces:

- conversation
- dossier artifact
- evidence / gap references

The user should not feel like they are switching apps.
These are shifts in emphasis inside the same room.

---

## War Room top bar

War Room can support slightly more visible controls than Front Porch, but only a few.

Recommended top bar:

### Left

- Helios mark
- engagement / room title

### Center

- optional lightweight room state
  - e.g. `Working`, `Reviewing`, `Preparing delivery`

### Right

- `Back to Front Porch`
- `Recent`
- `Share` or `Export`
- profile / workspace menu

Optional but restrained:

- one dropdown for view emphasis
  - `Trail`
  - `Collection`
  - `Graph`
  - `Delivery`

Avoid:

- large multi-icon action bars
- admin toolbars
- permanent environment controls

---

## Alerts, gaps, and opportunities

War Room is where these become visible, but they must be phrased like intelligence, not software.

### Good labels

- `What changed`
- `What is weak`
- `What I can pull next`
- `What is blocking confidence`

### Bad labels

- `connector errors`
- `pipeline status`
- `ingestion warnings`
- `validation metrics`

The user cares about:

- what shifted
- what remains uncertain
- what should happen next

Translate everything into that language.

## Graph doctrine inside War Room

The graph belongs in War Room, but only under strict rules.

### The graph is allowed to do three things

1. explain a relationship path
2. expose a hidden structure
3. support a specific finding

### The graph is not allowed to do these things

1. dominate the room by default
2. serve as decorative intelligence theater
3. replace the narrative

### Ideal graph usage

- AXIOM says something
- the user asks why
- the graph appears as the shortest useful explanation

Not:

- the graph is always there because “we have a graph”

---

## Dossier doctrine inside War Room

War Room should not force the user out of the room to consume the deliverable.

The dossier should appear as:

- a live briefing artifact
- a readable structured object
- expandable and discussable in place

War Room should support:

- viewing a dossier section
- asking AXIOM to challenge or expand that section
- tracing a claim back to the evidence
- marking a section as needing deeper work

This turns the dossier from output into collaboration substrate.

## Anti-patterns

War Room fails when it becomes:

### 1. Dashboard software

Symptoms:

- too many equal-weight cards
- KPI tiles everywhere
- grid-first layout
- the user scans modules instead of following the thread

### 2. Tool cockpit

Symptoms:

- too many buttons
- too many always-on controls
- obvious backend features in the foreground

### 3. Graph theater

Symptoms:

- giant graph dominates the room
- little narrative context
- “cool” but not decision-relevant

### 4. Chat bolted onto analytics

Symptoms:

- chat is a small side widget
- the real UI is still a dashboard
- AXIOM feels secondary

### 5. Faux military aesthetic

Symptoms:

- aggressive threat-board visuals
- overly tactical styling
- dramatic effects without substance

War Room should feel serious.
Not performatively “intel.”

---

## Transition doctrine

The transition from Front Porch to War Room should feel intentional.

Good transition language:

- “I have the preliminary picture. If you want, we can open the War Room and pull on the ownership chain, challenge the confidence, or redirect collection.”

This matters because it tells the user:

- Front Porch was enough to begin
- War Room is for depth, not because the first room failed

War Room is an invitation to deepen the work, not a handoff to a different product.

---

## Does this meet Tye’s intent?

Every War Room design move should be tested against these questions:

1. Does this feel like collaborating with AXIOM rather than operating software?
2. Is the conversation still the center of gravity?
3. Does each visible pane earn its right to exist?
4. Is the room deeper and more serious without becoming cluttered?
5. Does the graph serve explanation instead of spectacle?
6. Would a sophisticated practitioner feel empowered rather than burdened?
7. Does the interface feel elite without feeling theatrical?

If the answer to two or more is “no,” the move is wrong.

---

## Final rule

Front Porch is where the user starts talking.
War Room is where the problem becomes vivid.

But in both rooms, the user should always feel like they are working with intelligence, not with software.
