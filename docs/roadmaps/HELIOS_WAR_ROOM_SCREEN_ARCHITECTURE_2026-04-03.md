# HELIOS_WAR_ROOM_SCREEN_ARCHITECTURE_2026-04-03

## Bottom line

War Room is a deeper room, not a dashboard.

It should feel like walking from a calm briefing space into an active intelligence workspace where AXIOM is still leading, but the trail, evidence, and challenge surfaces are now visible.

The user should feel:

- the same case continued
- higher fidelity
- more seriousness
- more agency
- not more software

If War Room feels like a SaaS app with lots of panels, it has failed.

---

## Visual thesis

**Mood**: dark, focused, premium, serious  
**Material**: spatial surfaces around a live conversation and active finding  
**Energy**: controlled intensity, calm command, collaborative pressure

War Room should feel cinematic without becoming theatrical.

---

## Content plan

War Room needs four visible responsibilities:

1. hold the live collaboration with AXIOM
2. keep the active problem or finding in the center
3. expose evidence and provenance around the problem
4. let the user challenge, redirect, or deepen the work without operating machinery

Everything visible in the room must serve one of those jobs.

---

## Interaction thesis

War Room is still conversation-first, but the conversation is no longer alone.

The interaction model is:

- AXIOM advances the inquiry
- the user challenges or redirects when needed
- evidence, graph, and dossier fragments appear as live companions to the conversation
- switching focus should feel like glancing around a room, not changing apps

---

## The central rule

The center of War Room is always the work.

The center may be:

- the live AXIOM exchange
- an active finding
- a dossier fragment under challenge
- a graph trail under inspection

The center must never be:

- navigation chrome
- a KPI dashboard
- empty ornamental space
- settings

---

## Room anatomy

War Room has four structural zones.

### A. Room header

Minimal and recessed.
This is not a product nav bar.

#### Left

- Helios mark
- current case / engagement title
- tiny context marker like `War Room`

#### Center

- optional case status sentence
- optional quiet live status

#### Right

- `Front Porch`
- `Recent`
- share/export
- profile/workspace menu

Optional room-level tools can sit in a tucked control cluster, but the header should never dominate the room.

### B. Left context rail

This is not app navigation.
It is the problem-context rail.

It can hold:

- target summary
- object identity
- current mission framing
- open intelligence gaps
- recommended next threads

This rail should be narrow, informative, and collapsible.
It should never turn into a feature menu.

### C. Central intelligence canvas

This is the heart of the room.

It should hold:

- the live conversation
- the active finding
- the current working artifact

This area should feel large, breathable, and primary.
If the eye is drawn first to side rails, the hierarchy is wrong.

### D. Right evidence rail

This is where the proof lives.

It can contain:

- source excerpts
- provenance trail
- source-tier badges
- challengeable claims
- recent alerts
- graph path fragments

The right rail should be the room’s memory and evidence wall, not a generic details inspector.

---

## Recommended layout concept

### Recommended concept: Conversation-Centered Command Table

This is the strongest default.

#### Structure

- narrow left context rail
- broad center collaboration canvas
- medium right evidence rail

#### Why it wins

- keeps AXIOM central
- lets the user feel surrounded by the work without drowning in it
- preserves the emotional continuity from Front Porch
- gives evidence real presence without letting it hijack the room

#### Center canvas stack

The center canvas should be able to stack these in one vertical working thread:

1. AXIOM lead message
2. active finding or inquiry block
3. dossier fragment or graph fragment
4. user response
5. next AXIOM move

That thread should feel alive and editorial, not like a debug console.

---

## Alternate layout concepts

### Alternate 1: Artifact-Centered Studio

Best when the user is shaping or challenging a dossier artifact.

- artifact dominates center
- AXIOM conversation docks beneath or alongside it
- rails become secondary

Use this when delivery or narrative shaping is the active task.

### Alternate 2: Trail-Centered Investigation Room

Best when the user is following a knotty lead.

- center shows a graph path or evidence trail
- AXIOM floats adjacent to the trail
- right rail surfaces corroboration and contradiction

Use this only when the investigation itself is the active object.

Neither alternate should replace the default room.
They are task-weighted states within the same room.

## Visual language

### Background

- deep charcoal or slate field
- soft tonal variation
- almost no visible texture
- no noisy gradients

### Surfaces

- active surfaces slightly brighter than the room
- surface separation should come from tone and spacing before border weight
- glows should be rare and meaningful

### Accent

- one restrained accent family
- semantic colors only when tied to evidence, risk, conflict, or resolution

### Typography

- strong type hierarchy
- large but disciplined headings
- compact evidence text
- quiet labels

War Room should feel serious enough that typography carries much of the authority.

---

## Room states

### State 1: Entry from Front Porch

The user should feel they entered the same case at deeper fidelity.

Visible on arrival:

- current question or mission framing
- AXIOM’s latest working statement
- open gaps
- evidence rail already populated with the most relevant trail

### State 2: Live inquiry

The room is focused on a current question.

Center:

- AXIOM message
- user reply
- current working object

Right rail:

- trail excerpts
- related evidence

Left rail:

- target identity
- mission frame

### State 3: Challenge

The user disputes a claim, confidence level, or interpretation.

The room should support:

- challenge inline
- visible reasoning
- alternate hypothesis surfacing
- re-tasking AXIOM without leaving the room

### State 4: Collection redirect

The user asks AXIOM to pull a different thread.

The room should let AXIOM say things like:

`I can chase the lien-holder path, the incumbent teammate path, or the protest trail. The lien-holder path is most likely to clarify control.`

The room should support that interaction without turning into a collection dashboard.

### State 5: Delivery shaping

The user is helping shape or interpret a returned dossier.

The center should hold:

- dossier fragment
- AXIOM synthesis
- user instruction

The rails should support:

- underlying evidence
- related graph paths
- specific gap items

---

## Graph doctrine inside War Room

The graph is not the room.
The graph is a supporting visual instrument.

Graph should appear when it helps:

- explain a relationship path
- expose a hidden intermediary
- show team structure
- show lineage or control

Graph should not appear by default just because Helios has a graph.

When a graph view opens, it should still feel subordinate to the live inquiry.
The user should understand why it is visible now.

---

## Dossier doctrine inside War Room

The dossier is not a terminal output.
It is a working intelligence object.

In War Room, the dossier should be:

- inspectable
- challengeable
- expandable
- traceable to sources

The user should be able to move from:

- narrative claim
to
- supporting trail
to
- collection gap
to
- AXIOM redirect

without feeling like they changed tools.

---

## AXIOM behavior in War Room

AXIOM should speak differently here than in Front Porch.

It should sound:

- sharper
- more explicit
- more peer-level

Good War Room lines:

- `I traced the ownership chain through three Delaware LLCs and hit a wall at a Cayman entity. The UCC filings point to a likely controlling lender. I can pull that thread next.`
- `The subcontractor picture is still thin. The strongest next move is archive-and-diff plus protest mining.`
- `This claim is holding on two public signals and one weak inference. I would not lock it as settled yet.`

Bad War Room lines:

- tutorial language
- dashboard narration
- robotic explanation of system internals

---

## What must be visible

War Room can expose more than Front Porch, but not everything.

The visible set should stay narrow:

- live AXIOM conversation
- active case framing
- open gaps
- trail or evidence excerpts
- optional graph fragments
- dossier fragments
- challenge / redirect controls

That is enough.

---

## What must remain hidden

- connector administration
- full collection orchestration UI
- validation-gate implementation details
- raw database jargon
- product-module navigation sprawl
- unrelated admin surfaces
- KPI dashboards unrelated to the active inquiry

If those appear, the room becomes a tool instead of a collaboration space.

---

## Transition from Front Porch

The transition must feel like moving into the same engagement, not launching a different app.

Rules:

1. carry the user’s question forward visibly
2. carry AXIOM’s current framing forward visibly
3. open the room already populated with relevant evidence
4. do not make the user restate the problem
5. do not expose a blank workspace

The moment should feel like:

- same case
- deeper room
- more instruments

not:

- new product mode

---

## Motion

### Entering War Room

- room darkens
- the central inquiry sharpens into view
- rails appear as supporting structures, not flying panels

### Opening a trail

- evidence expands close to the claim it supports
- the user should feel continuity, not teleportation

### Opening graph context

- graph should reveal from the current claim or entity
- not replace the room abruptly

### Closing depth

- when the user returns to a simpler state, the room should calm down gracefully

---

## Mobile and constrained-width behavior

War Room on constrained widths should preserve the same hierarchy:

- center first
- evidence second
- context third

Recommended behavior:

- center conversation and active artifact remain primary
- rails become drawers or sheets
- graph opens as a focused modal canvas

Under no conditions should mobile War Room become a tiny dashboard with unreadable sidebars.

---

## Anti-patterns

Do not ship:

- dashboard-card mosaics
- left-side app navigation disguised as context
- rainbow status palettes
- giant chart blocks with no inquiry attached
- center-stage KPI strips
- graph by default
- debug-console language
- admin-like bulk controls

War Room should feel elite and useful, not technical and proud of itself.

---

## Does this meet Tye’s intent?

War Room meets the intent only if the answer to all of these is yes:

1. Does it feel like a deeper collaboration room rather than a dashboard?
2. Is AXIOM still the primary interaction layer?
3. Does the center clearly hold the live work?
4. Do the rails sharpen the inquiry instead of competing with it?
5. Can the user challenge or redirect the work without operating machinery?
6. Does the room feel darker and more serious than Front Porch without becoming theatrical?
7. Would a sophisticated practitioner want to stay in this room for an hour?

If any answer is no, War Room is not done.
